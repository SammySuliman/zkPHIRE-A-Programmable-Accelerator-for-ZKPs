SumCheck Accelerator

Programmable Hardware for Zero-Knowledge Proofs

Ashesh Kaji · New York University

01 · Problem Statement

Why ZKP Provers Are Slow

Generating zero-knowledge proofs is computationally intensive. The SumCheck protocol — a core building block of modern proving systems like HyperPlonk — often dominates execution time on CPUs, taking minutes to hours for realistic circuits. This motivates hardware acceleration.

🖥️

CPU: Minutes to Hours

Software provers like Plonky3 run SumCheck entirely on the CPU. Multi-field multiplication dominates, and even with SIMD optimization, large circuits can take prohibitive time.

Bottleneck: O(d · 2μ) field multiplications per round
⚡

This Work: FPGA @ 200 MHz

We implement a SumCheck hardware accelerator on FPGA with 8 parallel Processing Elements (PEs) and a 16-bank scratchpad memory subsystem. Each PE contains a Montgomery modular multiplier feeding an extension engine, achieving paper-parity with the zkPHIRE ASIC design.

Target: 8 PEs × 1 multiply/cycle @ 200 MHz = 1.6 Gops/sec
02 · Architecture

SumCheck Accelerator Dataflow

Eight Processing Elements operate in parallel, fed by a 16-bank scratchpad memory over AXI-Stream interfaces. Each PE implements the full SumCheck data path: Update → Extension Engine → Product Lane → Accumulator.

03 · Core Arithmetic

Montgomery Modular Multiplication

At the heart of each PE lies a Montgomery multiplier — the most area-efficient approach for modular arithmetic on FPGAs (eliminating expensive DSP-based division). The three-step reduction computes a·b mod p using only shifts and additions.

1
Integer Product
T = a · b   // 512-bit intermediate
2
Reduction Factor
m = (T · p') mod R   // R = 2256, p' = precomputed
3
Montgomery Reduction
t = (T + m · p) >> 256   // final correction if t ≥ p
💡 Key insight: The Montgomery reduction replaces expensive division-by-p with a cheap multiplication and right-shift by 256 bits — perfectly suited for FPGA DSP slices configured as 18×25 multipliers.

08 · Limitations

Honest Assessment

Transparent discussion of current constraints and tradeoffs.

⚠️

Degree Limit: 16

The scratchpad register depth constrains maximum polynomial degree to 16, compared to the paper's degree-31 support. This limits the circuit complexity that can be handled in a single round without decomposition.

🔬

Single-Round Scope

We implement one SumCheck round; the full HyperPlonk prover requires ZeroCheck, PermCheck, and OpenCheck sub-protocols. Integration with the complete Plonky3 stack remains future work.

📐

FPGA vs ASIC Scale

FPGAs trade ASIC-level density for rapid prototyping. Our PYNQ-Z2 implementation saturates DSP resources at 8 PEs, but RFSoC-class boards offer clear headroom for scaling.

09 · Takeaways

Conclusion

Key contributions of this project.

📐

Paper-Parity FPGA Implementation

We successfully implement the zkPHIRE SumCheck accelerator architecture on FPGA, matching the paper's 8-PE structure, scratchpad banking, and Montgomery arithmetic engine.

🧮

Montgomery Eliminates DSP Bottleneck

Montgomery modular multiplication avoids division-hardware in DSP slices, enabling efficient use of available FPGA resources and reducing critical path delay.

🔗

Scalable 8-PE Architecture

The parallel PE design demonstrates linear scaling potential. RFSoC-class boards support higher PE counts and clock frequencies, promising competitive proof generation throughput.
