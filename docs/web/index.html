Paper: zkPHIRE: A Programmable Accelerator for ZKPs

The paper’s idea: programmable SumCheck for high-degree gates.

Instead of a fixed SumCheck datapath for one gate type, zkPHIRE supports arbitrary high-degree, expressive gate polynomials.

Programmability

Schedules different polynomial terms, degrees, and reuse patterns.

Locality

Uses tiled scratchpads to reuse MLE table data across terms.

System integration

Embeds SumCheck into a HyperPlonk accelerator with MSM, SHA3, modular inverse, and multifunction forest units.

09
Engineering design choice 1: validation-first development

The golden model is the specification.

Before optimizing hardware structure, we locked down one-round behavior in a bit-exact Python reference model and made every HLS target prove that it matches.

Reference semantics

golden_sumcheck.py defines BN254 field arithmetic, round polynomial samples, challenge-based table updates, and protocol-chain invariants.

Stable regression cases

We use hand-checkable deterministic examples plus randomized degree-2 through degree-6 checks to catch both math mistakes and indexing mistakes.

Hardware acceptance test

C-simulation is not considered passing unless every samples[x] value and every updated[m][k] entry matches the golden output.

\[ \text{HLS implementation is correct} \iff \forall x,m,k:\; samples_{hls}[x]=samples_{gold}[x] \land updated_{hls}[m,k]=updated_{gold}[m,k] \]

Engineering design choice 2: IP interface contract

We kept control scalar and moved field tables through memory.

The SumCheck round has tiny configuration state but large 256-bit table buffers, so the interface separates AXI-Lite control from bulk memory movement.

AXI-Lite for control

FieldWhy it is scalar
degreeSets number of MLE factors and \(d+1\) samples.
sizeCurrent power-of-two table length.
rVerifier challenge for next-table update.
returnStatus code for bad degree, size, or challenge.

BRAM array top

sumcheck_round_array gives a simple, structured C-simulation interface: tables[m][i], samples[x], and updated[m][k].

m_axi DMA top

sumcheck_round_axi flattens buffers for host integration: inputs at m*size+i, outputs at x and m*(size/2)+k.

Why this choice

It matches how a host would launch reusable IP: configure once, stream or DMA table buffers, then poll status and read deterministic outputs.

11
Engineering design choice 3: self-checking testbenches

Our testbenches check behavior, not just completion.

The HLS testbench is intentionally deterministic and element-wise, so failures point to the exact sample or updated-table entry that diverged from the golden model.

Hand-checkable cases

Case A uses \(x_1x_2x_3\) with challenge chain \((5,7,11)\), including edge checks for \(r=0\) and \(r=1\).

Composition coverage

Case B tests individual terms \(x_1x_2\) and \(x_2x_3\), then verifies the combined polynomial \(x_1x_2+x_2x_3\).

Element-wise assertions

Each run compares every round sample and every next-table cell; a passing run means the full observable IP contract matched.

1 · Load vector

Populate 256-bit MLE tables with deterministic fixtures.

2 · Run IP

Call the same top-level function used for synthesis.

3 · Compare

Check samples and updated against expected golden values.

4 · Report

Emit named PASS/FAIL cases; current suite reports 7/7 pass on both targets.

Engineering design choice 4: divide the paper into HLS modules

We translated the SumCheck datapath into small, testable header files.

Paper concept

Each zkPHIRE datapath stage became a named HLS unit instead of one monolithic kernel.

Debug path

Small headers made it easier to isolate arithmetic, extension, product, accumulation, and update bugs.

Reuse path

The top file composes modules into BRAM and m_axi interfaces without changing semantics.

13
Engineering design choice 5: prompt one module at a time

We used careful AI prompting as a hardware-specification tool.

Why prompts worked

Each prompt gave the AI a role, the mathematical contract, Vitis HLS constraints, resource notes, and the exact functions to implement.

Guardrails

We asked for one header at a time, kept interfaces stable through types.hpp, and validated the generated code with the golden model and C-sim testbench.

Example shown

This prompt targets field_arithmetic.hpp, which depends on types.hpp for the BN254 type and field prime.

Task: Write a header file field_arithmetic.hpp that implements modular arithmetic for the BN254 scalar field. Hardware bring-up issue: PYNQ-Z2 DSP budget

The design was functionally correct, but too DSP-heavy for the board.

Our PYNQ-Z2 C-simulation passed, and timing met the 100 MHz target, but the board only has 220 DSPs. Direct 256-bit BN254 modular multiplication pushed the implementation over that limit.

220

DSPs available on PYNQ-Z2

233

DSPs after shared-multiplier optimization

105%

DSP utilization: cannot comfortably fit on-board

What caused the issue?

The Phase 1 multiplier used ap_uint<512>(a) * b % FIELD_P. This is simple and bit-exact, but expensive for a small Zynq-7020 FPGA.

What we tried / learned

Sharing mod_mul reduced DSP use from 458 to 233, a 48% cut. Barrett reduction made DSP worse at 683, and BIND_OP impl=fabric was unsupported on xc7z020.

15
Conclusion: what the PYNQ issue taught us

Montgomery multiplication is the right fix for small FPGA deployment.

The PYNQ-Z2 failure was not a SumCheck correctness problem. It was an arithmetic architecture problem: generic modular division for BN254 multiplication consumed too many DSPs.

What Montgomery changes

Instead of computing (a*b) % p with a costly divider, operands live in the Montgomery domain. Multiplication becomes a 512-bit product, a reduction factor from the low 256 bits, a shift, and one conditional subtraction.

Why it helps PYNQ

The plan replaces the DSP-heavy % reduction in field_arithmetic.hpp with precomputed Montgomery constants: R=2^256, R^2 mod p, and p'=-p^{-1} mod R.
