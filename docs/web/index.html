zkPHIRE

A programmable FPGA accelerator for the SumCheck bottleneck in zero-knowledge proofs.

Sammy Suliman Jui-Teng Huang Ashesh Kaji
02
Zero-knowledge proofs, in one sentence

Prove the claim. Hide the witness.

A prover convinces a verifier that a statement is true without revealing the private information that makes it true.

Completeness

If the claim is true and both parties follow the protocol, the verifier accepts.

Soundness

If the claim is false, a cheating prover should not be able to convince the verifier.

Zero knowledge

The proof leaks nothing beyond the truth of the statement.

03
How computation becomes algebra

Modern ZK systems turn circuits into polynomial identities.

\[ f_{\text{Plonk}} = q_Lw_1 + q_Rw_2 + q_Mw_1w_2 - q_Ow_3 + q_C \]

A gate is valid when its polynomial constraint evaluates to zero. Instead of re-executing every gate, the verifier checks randomized polynomial claims.

For large computations, the prover still does the heavy work: evaluating, multiplying, summing, committing, and opening many finite-field polynomials.

04
The protocol kernel we accelerate

SumCheck reduces a huge sum to a few random checks.

The prover claims a value for a sum over the Boolean hypercube:

\[ C = \sum_{x_1,\ldots,x_\mu \in \{0,1\}} f(x_1,\ldots,x_\mu) \]
1 · Claim

Start with the claimed total sum.

2 · Round poly

Prover sends a univariate polynomial \(s_i(X_i)\).

3 · Check

Verifier checks \(s_i(0)+s_i(1)\).

4 · Challenge

Verifier samples random \(r_i\).

5 · Repeat

Fix one variable and halve the table.

05
Why this belongs in a hardware design class

Proof generation is the bottleneck.

The zkPHIRE paper reports that CPU proof generation can take minutes to hours. SumCheck is attractive because it has linear-time prover complexity, but it still performs enormous finite-field workloads.

256

BN254 scalar-field elements are 256-bit values, so multiply/reduce is expensive.

d+1

A degree-\(d\) SumCheck round needs \(d+1\) univariate samples.

2^\mu

MLE tables grow with the number of variables, creating memory bandwidth pressure.

06
Paper: zkPHIRE: A Programmable Accelerator for ZKPs

The paper’s idea: programmable SumCheck for high-degree gates.

Instead of a fixed SumCheck datapath for one gate type, zkPHIRE supports arbitrary high-degree, expressive gate polynomials.

Programmability

Schedules different polynomial terms, degrees, and reuse patterns.

Locality

Uses tiled scratchpads to reuse MLE table data across terms.

System integration

Embeds SumCheck into a HyperPlonk accelerator with MSM, SHA3, modular inverse, and multifunction forest units.

07

Large speedups, small proofs, and custom gate support.

08
Our implementation target

We implement one SumCheck round as a reusable HLS IP block.

This keeps the project aligned with the class requirements: a defined accelerator, a host interface, testbench, synthesis data, and reproducible documentation.

Operation implemented

  • Read MLE table pairs \((f_i[2k], f_i[2k+1])\)
  • Extend each pair to sample points \(z=0,\ldots,d\)
  • Multiply all factors in each product term
  • Accumulate \(s(z)=\sum_k P_k(z)\)
  • Update tables with verifier challenge \(r\)
09
Datapath for one product term

Pair → extend → multiply → accumulate → update.

\[ L_{i,k}(z) = f_i[2k] + \big(f_i[2k+1]-f_i[2k]\big)z \quad P_k(z)=\prod_i L_{i,k}(z) \]
10
Hardware architecture

Two targets: small-board baseline and RFSoC parallel design.

The RFSoC version demonstrates paper-style parallelism with adaptive processing elements, scratchpad banking, and a final reduction tree.

PYNQ-Z2 baseline

Single PE, BRAM and m_axi interfaces, functionally verified but DSP-constrained.

RFSoC 4x2 design

8 adaptive PEs, 16 scratchpad banks, per-PE sample vectors, tree reduction, and orchestrator-level table update.

Shared kernel

Both targets use the same SumCheck semantics and golden reference.

Host/IP interface

Clean contract: configure, start, poll, read outputs.

AXI-Lite control

FieldMeaning
degreeNumber of MLE factors; emits \(degree+1\) samples.
sizePower-of-two table length before this round.
rVerifier challenge, \(0 \le r < p\).
returnStatus: OK or invalid input code.

Memory movement

  • tables[m][i]: input MLE tables
  • samples[x]: output round polynomial samples
  • updated[m][k]: next-round halved tables
  • m_axi variant uses flat host buffers for DMA-style integration

14
What we learned

The hard part is not just math. It is mapping irregular math to regular hardware.

SumCheck is simple to write as equations and subtle to schedule as hardware.

Arithmetic dominates

Naive 256-bit modular reduction quickly consumes DSP budget on small FPGAs.

Memory structure matters

Banking and scratchpad placement determine whether PEs can stay busy.

Control placement matters

Updating tables once at the orchestrator avoids per-PE overwrite bugs.

Verification needs granularity

Checking every sample and updated table entry made debugging tractable.

15
Conclusion

A working accelerator slice for a real ZKP bottleneck.

We built and verified a programmable SumCheck-round IP block inspired by zkPHIRE, with a clean host interface, golden-model validation, HLS synthesis reports, and both PYNQ and RFSoC implementations.

Correct

Bit-exact against Python reference; 7/7 C-sim pass on both targets.

Integrated

BRAM and m_axi interfaces; IP export artifacts recorded.

Honest

RFSoC resources fit; PYNQ DSP and RFSoC clock are documented limitations.
